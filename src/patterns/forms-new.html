Designing digital forms

Use this guide to help you design digital forms for Co-op.

It includes guidance about:

  • understanding the purpose, structure and content of your form before you start
  • using form inputs: any point that we ask users for data
  • using form patterns: common ways of asking users for data

Know why you’re asking each question and what it’s needed for

Before you ask a user to answer a question, make sure that you know exactly how you will use the answer and why the answer is necessary for the service. If it’s not necessary, do not ask. The more you ask of the user, the more effort it takes them and the more likely they are to drop out of the service.

Tell users why we’re asking

When you ask for information tell users why we are asking for it and what we’ll do with it. This will not only increase trust but can affect the answer a user gives. For example, asking someone for an address to verify who they are, and asking for an address to deliver something to, could result in different responses from the user.[could show wills address example?]

Use simple, direct language

Do not burden a user with new terminology and concepts if you can avoid it. It increases the effort it takes the user to get through the form, and makes it more likely that they’ll stop using it and go elsewhere.

If the answer the user needs to give relies on their understanding of a complex concept, you must explain in plain English [wills executor example?]. Do not assume any prior knowledge of the subject you’re writing about. Do not assume a certain level of literacy, education or digital experience. Design for everyone[link to principle].

Even if you’re writing for a specialist audience, research shows that the higher the level of expertise the greater the preference for plain English.

Content that’s easy to understand benefits everyone.

Give information in context

Think about the most effective place for your user to get information. A website is not always the best channel.

Content should be timely: given at the point it’s needed. Do not give information within a form that a user needs to refer to at a later date. And do not disturb the task the user is completing – the form — by including a link within it that takes the user away from it.

Get information in context

If the user needs to understand complex information in order to answer a question, give that information at the point it’s needed. Allow users to easily refer back to it, re-read it or skip to parts they’re unsure of. Consider users who have short term memory, cognitive problems or are in a distracting environment — don’t expect complex information to be retained easily or for a long time. Make complex questions as easy as possible for your users to answer. See how the Wills Team tried to make writing a will less daunting.

And let users focus — only deal with one concept per page.

Set expectations

Tell users what’s expected of them before they start the form. It will reduce frustration and make it less likely that they’ll drop out of the transaction.

Tell users upfront:

  • what the service is
  • what the eligibility is
  • what information they need which they may not have immediately to hand
  • any related costs or commitments
  • any other essential information that may affect their decision to use your service

Form inputs

Form inputs are any place in your form where you ask the user for data. They can include:

  • radio buttons and checkboxes
  • text entry fields (known as ‘inputs’ and ‘text areas’)
  • drop down menus
  • submit buttons

Radio buttons and checkboxes

Radio buttons and checkboxes allow users to select an answer from 2 or more options.

Use radio buttons when the user can choose only one answer from a list.

Use checkboxes when the user can choose multiple answers from a list.

If you’re using either radio buttons or checkboxes, you must:

  • style them in a way that makes it clear which option or options has been selected
  • allow users to click anywhere in the box
  • display the options a user can choose from top to bottom, not left to right — this makes it quicker for users to scan

Where possible, give users the option to indicate that none of the options apply to them. Some users need this to be an explicit option to know that they can move on.

Text entry fields

If you need your user to type information into your form, use a text entry field.

If you’re using text entry field, you must:

  • give them a label: ‘Full name’, ‘Date of birth’, ‘Address’ and so on [link to labelling elements]
  • allow users to click on the label to let them to start answer (clicking the label takes the cursor to the text entry field)
  • don’t use placeholder text (text that sits within the form field) — if a question needs extra clarification, use hint text instead

Types of text entry field

Depending on the answer you’re expecting, use either:

  • a one line entry box (known as a ‘text input’) for things like name, email and so on
  • multiple line entry box (known as ‘text area’) for things like feedback, additional information and so on

Text input

With a text input you must also make the length of the box appropriate for the amount of information needed: do not make it frustratingly short or intimidatingly long.

Text area

Use large text areas with caution. Many users find it hard to type, do not like to type or worry about articulating themselves. Large text areas can be daunting and discouraging.

If you have reason to use them, research them with mobile device users. Entering information using touch devices can be especially time-consuming.

Text areas should:

  • scale appropriately on mobile devices
  • be a size that allows the user to scroll outside of it
  • have ‘draggable’ corners so users can control the size of the text area
  • show a countdown of remaining characters (if you’re restricting the characters that can be typed in the text area)

Drop down lists

Drop down lists (also known as ‘select inputs’) can be hard to use because they:

  • hide content from the user
  • can slow users down
  • are less accessible than other options

There are alternatives to using a drop down list. Try:

  • checkboxes or radio buttons for short lists
  • type aheads (the user is offered a list of relevant suggestions as they type), such as Baymard's Country Selector, for longer lists such as country and county lists
  • text entry fields if you do not need the answer to be matched with a back-end system

Only use a drop down lists if you have thorough research to support your decision.

Our recommendations on drop down lists is based on research by the Nielsen Norman Group and Government Digital Service.

Submit buttons

A submit button allow the user to send their data to us.

Submit buttons must:

  • be descriptive of what they’re doing, for example, ‘send order’
  • only appear at the end of your form
  • be the primary button on the page

If you’re designing a multi-page form, make the submit button look different to show users that they are at the end of the form.

Form patterns

Labelling elements

Asking for name and title

Only ask for name and title if it’s essential.The format of people’s names can vary. Some do not fit into conventional UK formats. It can be unnecessary and uncomfortable for users to define themselves using a title, or a separate first name — surname format.

Never gather a title to assume gender.

Find out how we ask for names and titles.

Asking for gender and sex

Don’t ask.

As with gathering a ‘title’, some users may feel uneasy giving this information.

If you do need to know, ask about a person’s ‘gender’, not their sex. Allow them to tell us in their own words with a text entry field

If you’re gathering information to use it to address the user later, ask what pronoun they prefer - he/him, she/her, they/them.

But ultimately - if you do not need to know - do not ask.

Validation

We should be building services that are flexible enough to allow users to give data in a way that they would naturally. For example we should allow users to enter a postcode in upper or lower case, with or without a space.

Validation errors should only be triggered on mandatory fields. They must be factual and help the user move on. [Link to content style guide]

Address

Only ask for a user’s postal address if it is essential information we need to fulfil the service. Telling users how you will use their address may affect their answer: their delivery address may be different to an address we need to verify their identity.

Example address details in wills

Progressive reveals

You might need to ask users for more information because of how they answered a previous question. You can use a ‘progressive reveal’ to do this so you only show the extra form field to the users who need to see it.

Only use progressive reveals to ask for one piece of information. If you need to ask for more than one thing, do so on a separate screen.

Example wills choosing executors

Multi-page forms

Only ask users for one type of information on screen at a time. This helps to keep them focused and makes the purpose of each page clear.

Continue and back

Allow users to move through the form with a ‘continue’ button and a ‘back’ link. Make the continue button the primary button on the page to encourage users to progress through the form.

Position your continue button on the bottom left of the page so it’s easier for users to see.

‘Continue’ buttons should always be active. If a user tries to progress without completing the necessary information, run [validation messages] to tell your user what the error is and how to fix it.

Progress trackers

A progress tracker gives your users an indication of how far through a form they are. But they can be more problematic than helpful — they can get in the way and confuse users.

Only use a progress tracker if you have thorough research to support the decision.
